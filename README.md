# Конвертер ресурсов Space Rangers HD

Набор конвертеров для форматов ресурсов **Space Rangers HD: A War Apart** —
распаковка, декомпиляция и обратная сборка без GUI, из командной строки.

| Формат | Что делает | Обратно |
|---|---|---|
| **`.scr` → `.rson`** | Декомпиляция бинарных скриптов в текстовый формат визуального редактора (группы, состояния, диалоги, переменные, код, тексты) | через `RScript.exe` |
| **`.pkg`** | Распаковка / упаковка архивов ресурсов (замена pack/unpack из ResEditor) — RAW и ZL02/zlib | встроено, byte-exact |
| **BlockPar `.dat` ↔ `.txt`** | Конвертация `CacheData`/`Lang`/`Main` и т.п. | через `BlockParEditor.exe` |

Ядро — на чистом Python (только стандартная библиотека). Внешние `.exe`
нужны лишь для части операций (см. ниже); распаковка `.pkg` и декомпиляция
`.scr` работают без них.

## Что понадобится

- **Python 3.9+** — больше ничего для `.pkg` и декомпиляции `.scr`.
- Опционально, модинг-утилиты сообщества (в репозиторий не входят):
  - `RScript.exe` — компилятор `.rson` → `.scr` (для контрольной пересборки `--check`);
  - `BlockParEditor.exe` — конвертация BlockPar `.dat` ↔ `.txt`.

  Положи их рядом со скриптами (`RScript_4.10f/RScript.exe`,
  `BlockParEditor_1.9/BlockParEditor.exe`) или укажи путь флагами
  `--rscript` / `--blockpar`.

## PKG — архивы ресурсов

Замена pack/unpack-части GUI-утилиты **ResEditor**: распаковка и сборка
`*.pkg` из командной строки. Поддержаны оба типа записей — RAW и ZL02 (zlib).
Перепаковка неизменённого архива даёт **байт-в-байт идентичный файл**, поэтому
собранные архивы гарантированно читаются игрой.

```bash
python run.py pkg list   DATA/english.pkg              # содержимое архива
python run.py pkg unpack DATA/english.pkg out_dir/     # извлечь в каталог
python run.py pkg pack   out_dir/ new.pkg              # собрать (zl02-сжатие как в оригинале)
python run.py pkg pack   out_dir/ new.pkg --raw        # собрать без сжатия
python run.py pkg verify DATA/english.pkg             # round-trip самотест
```

Типовой сценарий массовой обработки: `unpack` архива → правка файлов на диске →
`pack` обратно, без единого клика в GUI. Конвертация самих изображений внутри
(`gi/gai/hai` ↔ png) пока не входит — только перенос файлов в/из контейнера
без потерь (план по gi/gai/hai — в [docs/TECHNICAL.md](docs/TECHNICAL.md)).

Тот же инструмент доступен и как отдельный скрипт: `python srpkg.py <команда> …`.

## SCR → RSON — декомпиляция скриптов

Декомпилятор бинарных скриптов (`*.scr`) обратно в текстовый формат визуального
редактора **RSON** (`*.rson`). Восстанавливает группы, состояния, диалоги,
переменные, код и тексты — результат снова компилируется в `.scr` через
`RScript.exe`.

```bash
# Один файл — .rson появится рядом:
python run.py path/to/Mod_Foo.scr

# Каталог с модами (рекурсивно) — результаты в отдельную папку:
python run.py path/to/Mods --out-dir decompile_result

# С контрольной пересборкой через RScript.exe и побайтовой сверкой:
python run.py path/to/Mods --out-dir decompile_result --check
```

Для каждого скрипта в `--out-dir` создаётся папка:

```
decompile_result/
  Mod_Foo/
    Mod_Foo.scr          копия исходного бинарника
    Mod_Foo.rson         восстановленный RSON — открывается в редакторе
    Lang.txt / Lang.dat  тексты мода (если найдены рядом с .scr)
```

`Mod_Foo.rson` открывается в визуальном редакторе так же, как результат его
собственного декомпила `.scr`: объекты разложены по полотну, связи между ними
(корабль→звезда, группа→состояние, диалоговый узел→его код и т.д.) показаны
стрелками, добавлена информационная плашка с правилами. Переходы диалогов
(`DChange`/`DAdd`/`ChangeState`) остаются в коде как есть — ровно так же
поступает интерактивный декомпил самого редактора.

### Опции декомпилятора

| Флаг | Что делает |
|---|---|
| `--out-dir DIR` | Складывать результаты в `DIR/<Имя>/`; без него `.rson` кладётся рядом с `.scr` |
| `--check` | Пересобрать `.rson` обратно в `.scr` через RScript и сравнить с оригиналом |
| `--rscript EXE` | Путь к `RScript.exe` (по умолчанию ищется рядом) |
| `--blockpar EXE` | Путь к `BlockParEditor.exe` (по умолчанию ищется рядом) |
| `--timeout SEC` | Таймаут RScript на файл (по умолчанию 30) |
| `-v {verbose,brief,errors}` | Подробность лога |
| `--log FILE` | Дублировать лог в файл |

## BlockPar — `.dat` ↔ `.txt`

Обёртка над `BlockParEditor.exe` для конвертации BlockPar-файлов
(`CacheData`, `Lang`, `Main` и т.п.). Направление для одиночного файла
определяется по расширению.

```bash
python run.py blockpar path/to/Lang.dat            # -> Lang.txt рядом
python run.py blockpar path/to/Mod                 # батч: все *.dat -> *.txt (рекурсивно)
python run.py blockpar path/to/Mod --to-dat        # батч: все *.txt -> *.dat
```

## Частые вопросы (SCR → RSON)

**`SKIP … format version 6`** — очень старый формат скриптов, декомпилятор его
не поддерживает (встречается у единичных файлов).

**`check=MISMATCH`** — пересобранный бинарник отличается от оригинала. Небольшое
число таких файлов ожидаемо: оригинал был собран другой версией компилятора,
которая иначе нумерует плейсхолдеры `<N>` в текстах (подробности — в
[docs/TECHNICAL.md](docs/TECHNICAL.md)). На корректность декомпиляции это не влияет.

**RScript «завис»** — компилятор показывает модальные окна (в т.ч. безвредную
«Runtime error 217» при выходе). На Windows раннер сам находит такие окна,
нажимает OK и записывает текст ошибки в лог (`_dlgwatch.py`).

**Почему в коде диалогов ключи вида `CT("Script.Mod.5")`?** — это ссылки на
тексты из `Lang.dat`/`Lang.txt`. В сообщениях и ответах диалогов тексты
подставлены; в коде ссылки оставлены как есть — это нужно для точной
пересборки.

**Совпадают ли координаты объектов с интерактивным декомпилом редактора?** —
структура раскладки та же (переменные рядами сверху, колонка объектов слева,
состояние под своей группой, диалоги сеткой), но пиксель-в-пиксель координаты
могут отличаться: редактор считает ширину ячеек по метрикам системного шрифта,
и в `.scr` координаты не хранятся вовсе (это авто-раскладка). На точность
декомпиляции и пересборки это не влияет — только на внешний вид полотна.
