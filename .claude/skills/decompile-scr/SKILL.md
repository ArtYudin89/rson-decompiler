# Декомпиляция SCR → RSON

Скилл живёт в этом репозитории и запускает `run.py` из его корня. Пути ниже —
относительно корня репо (там, где лежат `run.py` и `decompiler.py`).

## Сторонние утилиты (в репо не входят)

Для `--check` (пересборка) и конвертации `Lang.dat` нужны модинг-утилиты
сообщества, которых в репозитории НЕТ (не распространяем чужие бинари):

- `RScript.exe` (RScript 4.10f) — компилятор `.rson` → `.scr`;
- `BlockParEditor.exe` — конвертация `Lang.dat` → `Lang.txt`.

`run.py` сам находит их, если положить рядом с собой:
`RScript_4.10f/RScript.exe` и `BlockParEditor_1.9/BlockParEditor.exe`.
Иначе путь задаётся флагами `--rscript` / `--blockpar`. Без них декомпиляция
работает полностью — пропускается только контрольная пересборка.

## Аргументы

Пользователь передаёт путь к `.scr`-файлу или каталогу с модами (рекурсивный
поиск). Если путь не передан — спросить или взять из контекста разговора.
Результаты по умолчанию класть в подкаталог `decompile_result` рядом с входом
(не в корень репо).

## Базовый вызов

```bash
cd <корень репо>
python run.py "<вход>" --out-dir "<выход>"
# если экзешники лежат не рядом с run.py, добавить:
#   --blockpar "<путь к BlockParEditor.exe>"
```

Каждый `.scr` даёт `<Имя>.rson` — восстановленный файл для визуального
редактора: объекты разложены по полотну, связи показаны стрелками
(`Visual.Links`), добавлена плашка. Переходы диалогов
(`DChange`/`DAdd`/`ChangeState`) остаются в коде — так же, как в собственном
декомпиле редактора.

## Проверка round-trip (только по запросу, только Windows)

```
python run.py "<вход>" --out-dir "<выход>" --check --timeout 10
# и --rscript "<путь>", если RScript.exe не рядом с run.py
```

RScript показывает модальные окна — раннер сам их закрывает (`_dlgwatch.py`),
вмешиваться не нужно. Прогон каталога с `--check` медленный (секунды на файл);
для больших каталогов запускать в фоне.

## Конвертация BlockPar: dat ↔ txt (только Windows)

Тем же `run.py` можно гонять BlockPar-файлы (`Lang.dat`, `Main.dat`,
`CacheData.dat` …) в текст и обратно — обёртка над `BlockParEditor.exe`
(так же, как `--check` оборачивает RScript):

```
python run.py blockpar "<файл>" ["<результат>"]
# направление — по расширению source: .dat→.txt, .txt→.dat
# --blockpar "<путь>", если BlockParEditor.exe не рядом с run.py

# батч по каталогу (рекурсивно, результат всегда рядом с исходным):
python run.py blockpar "<каталог>"           # все *.dat → *.txt
python run.py blockpar "<каталог>" --to-dat  # все *.txt → *.dat
```

Без `<результат>` файл кладётся рядом с исходным со сменой расширения. В батче по
каталогу `dest` не задаётся (каждый файл — рядом с исходным); битые/не-BlockPar
файлы идут в `FAIL`, остальные конвертируются. Модальные
окна ошибок раннер закрывает сам; при сбое — `FAIL` с текстом ошибки, без зависа.
Замечание: при `txt→dat` шифрование использует случайный seed, поэтому байты
`.dat` от прогона к прогону отличаются — это нормально (игра читает корректно,
round-trip по тексту совпадает).

## Как читать статусы

- `OK` — успех.
- `SKIP` — формат version 6, не поддерживается (норма для единичных файлов).
- `check=MATCH` — пересборка байт-в-байт; `check=MISMATCH@N` — расхождение
  (небольшой класс таких файлов ожидаем: version-diff компилятора, детали в
  `docs/TECHNICAL.md`); `check=NO_OUTPUT`/`check=rc…` — RScript не собрал
  (текст ошибки в скобках).

## Отчёт пользователю

Свести: сколько OK/SKIP/FAIL (и MATCH/MISMATCH при --check), где лежат
результаты. При FAIL — привести имена файлов и суть ошибок.
