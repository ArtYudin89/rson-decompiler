# Технические детали

Справка по формату `.scr`, устройству парсера, round-trip-пересборке и раскладке
полотна. Пользовательская инструкция — в [../README.md](../README.md).

## Состав репозитория

```
decompiler.py       ядро: парсер .scr -> структура -> .rson
run.py              CLI-раннер: батч, --check (round-trip), Lang.dat->Lang.txt
_dlgwatch.py        авто-закрытие модальных окон RScript.exe (Windows)
validate.py         сверка G/S/D-счётчиков с эталонными .rson
rsons/              эталонные .rson (ground truth для валидации)
Modding_Manual/     справка сообщества по формату/функциям скриптов
```

## Пайплайн

1. `decompiler.parse(data, lang)` разбирает бинарник в структуру
   (`Visual.Objects` + `Visual.Links`) и раскладывает объекты по полотну.
2. `decompiler.decompile()` пишет RSON (JSON с фиксированной шапкой
   `FileID`/`FileVersion`/`ScriptName`).
3. `run.py` для каждого `.scr`: находит `Lang.txt`/`Lang.dat` (вверх по дереву,
   `CFG/Rus/` и корень), при необходимости конвертирует `Lang.dat`
   BlockParEditor'ом, декомпилирует и опционально делает контрольную пересборку
   (`--check`).
4. При `--check`: RSON компилируется `RScript.exe --cli`, результат
   выравнивается бинарным патчером (см. ниже) и сравнивается с оригиналом
   побайтово.

## Формат бинарника

### Заголовок (первые 12 байт)

```python
h0, h1, h2 = struct.unpack_from('<III', data, 0)
```

| Флаг | Вывод | Смысл |
|---|---|---|
| `is_kavscr` | `h2 != 0` | «kavscr»-семейство: имя скрипта как первая переменная, `h1` = смещение секции переменных |
| `is_preglob` | `h2 >= 3` | `h2` = число pre-global переменных в шапке (не kavscr-инициализатор) |
| `is_new_fmt` | `h0 == 8` | «новая» структура записей (меньше паддингов и т.п.) |
| `is_nt_fmt` | `(h0 != 8) or (h2 == 0) or (h2 > 1)` | строки null-terminated; иначе — «rw»-строки |

- `h2 == 1` — классический kavscr; `h2 == 2` — kavscr с pre-global массивом
  (считается kavscr, не preglob); `h0 == 6` — старый формат, **не поддержан**
  (парсер сразу выдаёт SKIP).
- `h0` — версия формата *записавшего* компилятора: современный RScript всегда
  пишет 8, старые оригиналы несут 7 (метаданные, контент совместим).
- Файл **не выровнен глобально** по 2 байта — секции сдвигают выравнивание,
  поэтому UTF-16 можно декодировать только в локальных окнах от найденной точки.

### Строки

- nt-формат: null-terminated UTF-16LE — код хранится читаемым исходником, и
  правки, меняющие длину, безопасны (нет префиксов длины).
- rw-формат (kavscr `h2==1` при `h0==8`): строки с префиксом длины — любая
  правка внутри строки требует коррекции префикса.

### Переменные

Типы: `0=None, 1=Int(i32), 2=Dword(u32), 3=Float(f64), 4=Str, 9=Array`.
Массив (тип 9): `count` + `count × 3 байта` слотов `[idx16][type8]`; для слотов
с `type != 0` следом идут значения (int 4Б / f64 / строка). После основной
секции переменных могут идти «регистрации» объектных имён (`t=1/2` + dword) —
в частности, там живут настоящие приоритеты звёзд, когда в записях звёзд
дубликаты.

### Основные секции (по порядку)

Шапка переменных → звёзды → планеты → корабли → места (places) → предметы
(items) → группы → pre-state код-слоты → состояния → диалоги → сообщения →
ответы. Точный состав каждой секции зависит от комбинации `h0/h1/h2` и
количества звёзд; в `decompiler.py` ветвления прокомментированы значениями
заголовка, при которых встречен каждый вариант.

Ключевые особенности:

- **Мульти-звёздные скрипты**: у kavscr первая звезда лежит до планет,
  остальные — после кораблей (со своими планетами/кораблями/местами); у
  не-kavscr `h0=8` — полностью «перемежённая» раскладка (звезда → её планеты →
  её корабли, повторить). Вторичная звезда может нести +13 байт полей перед
  своим planet_count — решается peek-валидацией, не слепым скипом.
- **Pre-state код-слоты**: 5 wstring-слотов `[pad, pad, Init, Turn, DialogBegin]`
  (позиция слота определяет тип кода); в старых форматах — sentinel-цикл
  (`dword > 1000` = длина следующего чанка).
- **Ruins/StrengthMin/Max у кораблей и групп** (`h0=8` nt): 4 хвостовых
  wstring, реальное значение Ruins — в последнем, первый — паддинг.
- **Приоритет звезды** — знаковый: `0xFFFFFFFF` = «не задан» (→ 0), прочие
  отрицательные значения должны пройти round-trip как есть.
- **Item.+Place** может ссылаться не только на Place, но и на
  Group/Planet/Ship/Star — резолвится по всем секциям.

### Защитные сканы (scan-forward)

Счётчики секций местами оказываются мусором (следствие вариантов формата).
Парсер использует единый приём: откатиться к позиции счётчика и сканировать
вперёд до правдоподобной пары «маленький count + валидное имя следом»
(идентификатор с заглавной буквы). Жадные скипы нулей всегда гейтятся
контентной проверкой следующей записи (например, «это места или предметы?»),
а `_guard()` жёстко ограничивает каждый счётчик, чтобы мусор не приводил к
OOM-циклам.

## Канон выходного RSON

Выход выровнен по структуре, которую даёт интерактивный SCR-декомпил самого
редактора (и авторские сохранения):

- полная шапка редактора: `FileID … ScriptTextOut`, затем `LangDatIn/LangDatOut`,
  `MainDatIn/Out`, `CacheDatIn/Out`, `ExportLangTxt/ExportLangDat`,
  `Visual.Objects`, `Visual.Links`, `BlockPar.EC.*`. Поля `Export*` намеренно
  `false`, а `LangDatOut` пуст: непустой путь с `ExportLangDat=true` заставил
  бы редактор перезаписать `Lang.dat` мода при компиляции;
- порядок ключей узла: `Type, Name, Pos.x, Pos.y, Parent, #`, затем полезная
  нагрузка (у звёзд — сначала `Constellation`);
- пустой `Attack.Items` не пишется вовсе;
- плашка `Rect.Text` «SCR Decompiling &lt;имя&gt;» с правилами преобразования —
  как её добавляет интерактивный декомпил редактора.

### Раскладка полотна

Воспроизводит структуру, которую даёт интерактивный SCR-декомпил редактора
(**координаты в `.scr` не хранятся** — проверено: в хвосте бинарника код, а не
позиции; редактор раскладывает объекты детерминированным авто-алгоритмом):

- переменные — рядами сверху (шаг 25 по Y, мин. ячейка 60 по X);
- вертикальная колонка слева с шагом 45: корабль → звезда → ряд мест → планета →
  предмет (пустая секция слот не занимает);
- группы — компактной сеткой с переносом (шаг ряда 60, перенос по ширине);
- **каждое состояние — под своей группой** (`+State`-цель группы, `x, y+25`);
  состояния без группы-владельца — рядами ниже сетки;
- Turn-код узла (состояния, диалога, сообщения, ответа) — вплотную слева от
  владельца (`x−20`, тот же `y`);
- диалоги/сообщения/ответы — сетками-рядами (шаг 25);
- служебные опоры Global/Init/Turn/DialogBegin — внизу с шагом 60 по X.

Пиксель-в-пиксель совпадения с редактором нет: ширина ячейки у него зависит от
метрик системного шрифта (проверено — `PlayerState` и
`MachpellaWaitInTortugatzState` получают одинаковую ширину ячейки, а
`NeurolinksNormalState` — 900 px: это расстояния между поддеревьями графа, а не
ширина текста). Наша раскладка приближает структуру, а не точные координаты; на
декомпиляцию и пересборку это не влияет.

## Тексты: Lang.txt / Lang.dat / CT-ссылки

Тексты диалогов хранятся отдельно и адресуются ключами `Script.<Секция>.<N>`.
`Lang.txt` — плоский текстовый формат, парсится напрямую. `Lang.dat` сильно
зашифрован (энтропия ~7.96 бит; XOR/RC4/zlib не подходят) — конвертируется
только `BlockParEditor.exe --cli --convert`.

В сообщениях/ответах диалогов декомпилятор подставляет текст из Lang; в коде
операций CT-ссылки оставлены вербатим — это условие точной пересборки (см.
ниже).

## Round-trip (`--check`) и бинарный патчер

Прямой байт-идентичности при пересборке нет и быть не может:

- RScript **игнорирует dialogs-файл на входе** (только пишет свою таблицу
  ключей) и **ре-ключует все CT-тексты** в своём порядке обработки: дедупит,
  оборачивает строковые литералы в `CT()`/`Format(CT(k),"<0>",0)`, двойные
  обёртки `CT(CT(...))`, отбрасывает переменные в аргументах
  `DText`/`DAnswer`. Оригинальные разреженные ключи невоспроизводимы со
  стороны RSON.

Поэтому после компиляции работает патчер (`run.py`):

1. `_patch_dialog_codes` — выравнивание по вхождениям: найти в оригинале и в
   пересобранном бинаре спаны `CT(...)`/`DText(...)`/`DAnswer(...)` (по ссылке
   `"Script.<имя>."`), спарить их 1:1 по порядку и виду, и вернуть в
   пересобранный файл оригинальные байты спанов. Литералы, обёрнутые RScript'ом
   в CT, восстанавливаются из writeback-таблицы (`newkey=text`), которую RScript
   написал при компиляции. При любом рассинхроне пар — откат без изменений.
2. `_unwrap_literal_formats` — «дожим» по первому дифф-байту, циклом: развернуть
   `Format(<лит>,"<0>",0,...)` обёртки, выровнять нулевой паддинг и пробельные
   сдвиги (реиндентация кода), вернуть съеденные экранирования `\'`, восстановить
   отрицательные значения, зажатые в `-1`, цифры плейсхолдеров `<N>`,
   1-ulp дрейф float-полей и структурные вставки/удаления старых версий записей
   (со строгой ресинхронизацией по 24 байтам).
3. Финальный арбитр — побайтовое сравнение: ложный MATCH невозможен.

Ограничение патчера: он реконсилирует **подстановки**, а не **вставки** —
для вставок кода нужно редактировать UTF-16-исходник в бинаре напрямую
(nt-формат это позволяет: строки null-terminated).

Замечания:

- RScript падает «Runtime error 217» на выходе (безвредно) и показывает
  модальные окна с реальными ошибками — `_dlgwatch.py` находит окна процесса
  (Win32 `#32770`), жмёт OK и возвращает текст в лог; пустой Global-слот
  эмитится как `['//']`, иначе RScript падает.
- Оставшийся класс расхождений — version-diff: старый компилятор иначе
  нумерует `Format`-плейсхолдеры `<N>` (ренумерация `<1>`→`<0>` и т.п.),
  такие файлы дают стабильный MISMATCH при полной корректности контента.

## Стрелки диалогов остаются кодом

Важный факт, подтверждённый эталоном (интерактивный SCR-декомпил редактора,
`*_recovery.rson`): переходы `DChange(N)` / `DAdd(N)` / `ChangeState(x)` внутри
операций **остаются в коде вербатим**, редактор их в стрелки НЕ превращает. Он
рисует только «структурные» связи (`Visual.Links`): объект→объект
(корабль→звезда, планета→звезда, группа→планета, группа→состояние, место→звезда,
предмет→место) и диалоговый узел→его Turn-код. Ровно этот набор связей строит и
наш декомпилятор — то есть «стрелки» уже присутствуют в обычном `.rson`, и
отдельная «графовая» форма не нужна.

## Валидация

- `run.py --check` — основной критерий (побайтовый MATCH).
- `validate.py` — быстрая сверка счётчиков групп/состояний/диалогов с
  эталонными `.rson` из `rsons/` (авторские файлы редактора — золотой источник
  схемы полей).
- Полезный приём при отладке: у авторских раздач рядом с `.scr` бывают
  `_dump.txt` — вербатим-дампы бинарника (UTF-16), удобны для построчного
  диффа кода.

## Известные ограничения

- Формат `h0=6` (очень старые скрипты) не поддержан — файлы пропускаются.
- Небольшой класс файлов даёт `check=MISMATCH` из-за version-diff компилятора
  (см. выше) — содержимое декомпиляции при этом корректно.
- `--check` и авто-закрытие окон работают только на Windows (RScript — win32).
- Парсер ограничен таймаутом на файл (`PARSE_TIMEOUT`, 10 c) — зависшие
  разборы пропускаются с пометкой TIMEOUT.
